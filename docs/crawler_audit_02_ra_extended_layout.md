# クローラ監査 #02: RAレコード拡張レイアウトの馬場状態・ハロンタイム欠落

## 概要

近年のJRA-VAN JV-Data の RA レコード（レース詳細）は「拡張レイアウト」
（実測 1272 バイト）で配信される。`RAParser` の拡張レイアウト分岐が
入線頭数（field48, pos886）の直後で **+2 バイトずれ**ており、以下の
スカラーフィールドを一切正しく読めていなかった。

- 天候コード（field49, pos888）
- 芝馬場状態コード（field50, pos889）
- ダート馬場状態コード（field51, pos890）
- ラップタイム先頭（field52, pos891）
- 障害マイルタイム（field53, pos966）
- 前3/前4/後3/後4ハロン（field54-57, pos970-981）
- レコード更新区分（field59, pos1270）

従来コードは pos890（ダート馬場状態）を `TenkoCD` として読み、天候・芝/
ダート馬場状態・ラップ・ハロンタイムは 856 バイト互換位置（賞金配列の
断片）を読むため、**全レコードで `'0'` / `'000'` に化けていた**。

## 影響

- `nl_ra` の `sibababacd` / `dirtbabacd`（馬場状態）が全レコードで無効
  （`'0'`）。→ KPS 特徴量の `babajotai_code_shiba/dirt`、`hist_track_*`、
  `basic_track_condition_*` が死亡（feature 監査 LOOP7 の真因）。
- `laptime` / 前後ハロン（レース時計）が全レコードで無効。→ 上がり・
  ラップ由来特徴量（feature 監査 LOOP9 の `rating_speed_late_stamina`
  ほか）の上流ソース欠落。
- **`tenkocd`（天候）には +2 ずれで「ダート馬場状態」の値が入っていた**
  ため、KPS の天候特徴量も誤った値を参照していた。

## 根拠（権威ある仕様）

JV-Data 4.9.0.1 仕様書（`JV-Data4901.xlsx`, フォーマットsheet, 1-indexed）:

| field | 名称 | pos | 長さ |
|------:|------|----:|-----:|
| 48 | 入線頭数 | 886 | 2 |
| 49 | 天候コード | 888 | 1 |
| 50 | 芝馬場状態コード | 889 | 1 |
| 51 | ダート馬場状態コード | 890 | 1 |
| 52 | ラップタイム | 891 | 25×3 |
| 53 | 障害マイルタイム | 966 | 4 |
| 54 | 前3ハロン | 970 | 3 |
| 55 | 前4ハロン | 973 | 3 |
| 56 | 後3ハロン | 976 | 3 |
| 57 | 後4ハロン | 979 | 3 |
| 58 | ＜コーナー通過順位＞ | 982 | - |
| 59 | レコード更新区分 | 1270 | 1 |

field58（コーナー）の pos982（1-indexed）は既存の
`EXTENDED_CORNER_OFFSET = 981`（0-indexed）と一致しており、コーナー展開
ロジック（PR #122）はこの監査後も整合する。

## 実データ検証

生キャッシュ `var/staging/jra/raw_cache/nl/RACE/20250705.bin`（2025-07-05
函館 全36R、すべて 1272 バイト）に対し修正後オフセットで再パース:

- 天候: `1`（晴）/`2`（曇）— 実測妥当。
- 芝馬場: 芝レースで `3`（重）→`2`（稍重）（日中に回復）、ダートレースで `0`。
- ダート馬場: ダートレースで `4`（不良）→`3`（重）、芝レースで `0`。
- 全馬場コードが仕様範囲 `0-4` に収まる（修正前の推定オフセットで出ていた
  `7`/`9` 等の無効値は消滅）。
- 後3ハロン（上がり）は 36/36 レースで有効、ラップ 35/36 で有効。

## 修正

`src/parser/ra_parser.py` の拡張レイアウト分岐を仕様準拠のオフセットへ修正:

- `TenkoCD = data[887:888]`, `SibaBabaCD = data[888:889]`,
  `DirtBabaCD = data[889:890]`。
- `LapTime = data[890:893]`（base レイアウト同様に先頭1ハロンのみ格納。
  `nl_ra.laptime` が単一 TEXT カラムのため）。
- `SyogaiMileTime = data[965:969]`, `Haron3F/4F/3L/4L = data[969:981]`。
- `RecordUpKubun = data[1269:1270]`。
- 周回数（Syukaisu）は RA レコードでは ＜コーナー通過順位＞ 節内の
  フィールドであり、拡張スカラー位置には存在しない。ここでは初期化し、
  コーナー展開ロジックが設定する（従来の pos887 読みは天候の誤読だった）。

回帰テスト（`tests/test_ra_parser_jravan.py`）を仕様準拠へ更新し、
拡張レイアウトの天候/馬場/ラップ/ハロンを検証する専用テストを追加。

## 復旧可能性

これはパーサのバグであり、生キャッシュ（RACE レコード）の再 import により
**過去分も遡って復旧可能**（MING のような当日発表・前向き限定とは異なる）。

## 残課題（別 PR）

- ラップタイム全25ハロンの取り込み（`nl_ra` schema / importer /
  DuckDB materialize の拡張が必要）。
- 再 import による過去データの馬場状態・ハロンタイム復旧の実行（運用手順）。
- NAR パーサ（`jrvltsql-nar`）の同等レイアウト監査。
